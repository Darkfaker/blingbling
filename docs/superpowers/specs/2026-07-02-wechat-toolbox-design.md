## 0. 结论

bbl 不应被设计成“一个装了五个功能的小程序”，也不需要在 v0.1 伪装成运行时插件平台。它应是一个**个人工具工作台**：首页负责快速找到和恢复最近使用的工具；每个工具是一个有统一元数据、页面约定和数据边界的业务模块；新增工具通过模板和构建期注册完成。

v0.1 的目标不是功能数量，而是验证以下闭环：

1. 用户能在 10 秒内打开目标工具并完成一次核心操作。
2. 本地工具在退出、切后台和异常终止后能正确恢复。
3. 云端工具在弱网、重复提交和多设备使用时不丢数据、不产生明显重复。
4. 开发者能在 30 分钟内创建一个可运行的空工具，在 1 个工作日内完成一个与现有复杂度相当的工具。

首版保留 4 个一级工具：**待办、番茄钟、二维码、阅读**。原“资讯”和“收藏夹”合并为“阅读”，因为收藏是内容域中的行为，不是独立工具。这样仍覆盖本地状态、云端 CRUD、Canvas/文件权限和远端内容四类技术风险，同时减少一个割裂入口。

## 1. 重新分析需求

### 1.1 用户真正要解决的问题

目标用户不是在寻找一个公共“工具大全”，而是希望把自己高频、零散、容易遗忘的小需求放进一个长期可维护的入口。核心价值按优先级排序：

1. **到达快**：无需搜索公众号、聊天记录或多个小程序。
2. **继续上次状态**：待办、计时和收藏不因退出而丢失。
3. **添加新工具快**：统一骨架降低重复工程成本。
4. **可控**：数据范围清楚，依赖少，个人能够长期维护。

“插件化”“多端”“自建后端迁移”是实现偏好，不是用户价值。在没有第二个平台和第三方插件需求之前，不为它们支付复杂度。

### 1.2 用户与场景

| 用户 | 高频场景 | 需要 | 不需要 |
|---|---|---|---|
| 开发者本人 | 每日多次快速打开 | 最近使用、稳定恢复、低维护 | 新手引导、运营体系 |
| 熟人 | 偶尔使用单个工具 | 一眼看懂、无感身份、错误可恢复 | 复杂个性化、社交关系 |
| 维护者 | 新增或调整工具 | 模板、类型检查、自动校验 | 运行时下载插件 |

### 1.3 明确不做

- 不做运行时插件安装、第三方插件市场或远程代码加载。
- 不为尚不存在的多端需求选择或抽象公共后端。
- 不保存微信昵称、头像等非必要个人资料。
- 不做账号密码、独立登录页或强制授权弹窗。
- 不做待办协作、资讯运营后台、推送提醒和复杂统计。
- 不承诺小程序退到后台后持续精确计时；只保证恢复时计算正确。

### 1.4 成功指标与验收方法

| 指标 | 目标 | 验收 |
|---|---:|---|
| 首页到核心操作 | 中位数 ≤ 10 秒 | 5 名测试者各完成 4 个任务 |
| 工具可达性 | 首页至任意工具 ≤ 2 次点击 | 路由清单检查 |
| 状态恢复 | 4 个规定场景 100% 正确 | 前台、后台、杀进程、跨日测试 |
| 云端写入 | 失败可见、可重试、无明显重复 | 断网/超时/连点测试 |
| 新模块骨架 | ≤ 30 分钟 | 从模板创建并通过 CI |
| 稳定性 | 核心路径无 P0/P1 缺陷 | 真机测试矩阵 |

“五个工具全部可用”不是有效成功指标；工具数量不代表价值或架构质量。

## 2. 产品结构

### 2.1 信息架构

```text
首页
├── 继续使用（最近 1 个工具，可关闭）
├── 工具网格
│   ├── 待办
│   ├── 番茄钟
│   ├── 二维码
│   └── 阅读
└── 设置
    ├── 数据与隐私
    ├── 清理本地缓存
    ├── 反馈与版本
    └── 调试信息（仅开发版）
```

v0.1 不使用底部 Tab。只有 4 个入口时，单首页更快，也避免把工具错误地塑造成同级主导航。

### 2.2 首页

- 首屏展示应用名、最近使用卡片和 2×2 工具网格，不放资讯流。
- 点击工具卡直接进入，不增加二级分类页。
- 最近使用只保存 `toolId` 和访问时间；模块被移除时自动忽略无效记录。
- 工具不可用时展示明确状态，例如“内容源暂不可用”，而不是保留可点击的坏入口。
- 首页配置来自同一份模块清单，避免入口、路由和权限声明各维护一份。

### 2.3 通用交互状态

每个云端页面必须实现 5 种状态：首次加载、正常、空、可恢复错误、无权限。所有写操作必须提供进行中反馈并防止连续提交。错误文案说明用户下一步，不展示 SDK 原始错误。

## 3. 四个工具的规格

### 3.1 待办

**核心任务**：在 15 秒内记下一件事，并能完成或撤销。

v0.1 支持：新增、完成/撤销、编辑标题、删除、待办/已完成分组。标题 1–120 字，空白标题不提交。删除后保留 5 秒撤销入口。

数据策略：

- 云端为权威数据，本地保存最近一次成功快照。
- 首屏先显示本地快照并标记同步中，随后以云端结果校正。
- 新增使用客户端生成的 `id`，重复重试仍写同一条记录。
- 更新带 `updatedAt`；v0.1 冲突规则为“最后成功写入胜出”，并记录同步失败供重试。
- 不实现完整离线编辑队列。离线时允许查看快照；写入明确提示“联网后重试”，避免半套离线逻辑造成数据幻觉。

### 3.2 番茄钟

**核心任务**：开始一次专注，并在回到小程序时得到正确剩余时间或完成状态。

状态机：

```text
idle ──start──> running ──pause──> paused
  ^                │                  │
  └────reset───────┴────resume────────┘
                   │
                elapsed
                   │
              completed ──next──> running / idle
```

计时依据是绝对时间，不递减保存“剩余秒数”：

- 运行时保存 `startedAt`、`durationMs`、`pausedAccumulatedMs` 和当前阶段。
- 页面显示可用定时器刷新，但真实剩余时间始终由 `now - startedAt` 推导。
- `onShow`、冷启动和跨日恢复时重新计算；若已结束，只触发一次完成迁移。
- 暂停时保存已消耗时长；重启后保持暂停。
- 震动和 Toast 仅在小程序活跃时作为提示，不承诺系统级后台提醒。

默认专注 25 分钟、休息 5 分钟；允许 1–120 分钟调整并本地保存。v0.1 不上云、不做历史统计。

### 3.3 二维码

**核心任务**：把一段文本可靠地生成并保存为可扫描二维码。

- 输入支持文本和 URL，最大 500 字节；展示实时字节数而非只数 Unicode 字符。
- 输入采用 250ms 防抖，空输入展示示例态。
- 明确选择二维码纠错级别与边距，保存图片至少 1024×1024，白底、足够静区。
- 保存前在真机生成临时图片；相册权限被拒后展示前往设置的明确路径。
- 对过长内容给出限制提示，不生成可能无法扫描的结果。
- 发布前用微信扫码、iOS 相机、Android 相机各验证短文本、中文、长 URL。

具体二维码渲染库在 P0 spike 后锁定；以包体积、Canvas 2D 兼容、中文编码和真机保存成功为选择标准，不在设计阶段写死库名或未经测量的体积。

### 3.4 阅读

**核心任务**：浏览可信内容源，并保存稍后阅读条目。

- 列表、详情和收藏属于同一模块；“已收藏”是列表筛选，不是一级工具。
- v0.1 只接入 1 个明确许可且稳定的内容源；内容源未确定前，本模块不得进入正式开发。
- 云函数定时拉取并缓存，客户端只读缓存，不由每个页面加载触发抓取。
- 条目唯一键为 `sourceId + externalId`；保存标题、摘要、原文 URL、发布时间、抓取时间和必要的安全正文。
- 收藏只保存文章键与用户键，文章内容不复制进收藏记录。
- 抓取失败继续返回旧缓存，并展示“更新于 …”；没有缓存才显示错误。
- 外部链接和富文本需经过协议白名单与内容清洗；不直接渲染来源 HTML。

若 P0 结束仍无法确认内容源的合法性、稳定性和调用限制，则 v0.1 移除阅读模块，以“文本暂存/剪贴板工具”替代。这个止损点比先写一个泛化 RSS 抓取器更重要。

## 4. 技术方案

### 4.1 技术选型

推荐 **Taro + React + TypeScript + Sass + 微信云开发**，但理由是团队已有 React/TS 能力和组件复用效率，而不是尚未排期的多端发布。

采用前必须在 P0 用真机 spike 验证：

1. 当前锁定版本能稳定编译微信小程序。
2. Canvas 生成与保存二维码正常。
3. 云开发身份、数据库权限和云函数可用。
4. 冷启动包体积和首屏时间可接受。

如果其中任一核心能力需要大量平台补丁，退回**微信原生 TypeScript**。框架是可替换决策，不是项目目标。

不采用 Taro UI 作为默认组件库。该项目组件数量少，优先维护一组轻量业务组件，减少版本耦合与样式覆盖成本。需要复杂组件时再逐项引入并记录体积。

### 4.2 架构边界

```text
pages（页面入口，必须在构建配置声明）
  └── features（按业务能力组织）
        ├── UI / state / schema
        └── repository interface
              ├── local repository（Storage）
              └── cloud repository（Database / Function）

shared
├── ui（Button、Card、StateView、Toast）
├── platform（storage、canvas、network、lifecycle）
├── observability（错误归一化、日志）
└── utils（纯函数）
```

只在确有两个实现时建立接口，例如待办的本地快照和云端仓储。删除 `externalApi.ts` 这类“未来也许有用”的空抽象。CloudBase 迁移不是只改一个文件：身份、权限语义、数据模型、事务和运维都会变化，应在真正迁移时以 repository 边界处理。

### 4.3 模块清单：构建期扩展，不是运行时插件

微信小程序页面需要进入全局页面配置，因此模块不能在运行时动态注入路由。使用一个类型化清单作为入口元数据，并由构建期脚本校验页面配置：

```ts
type ToolId = 'todos' | 'pomodoro' | 'qrcode' | 'reading'

interface ToolManifest {
  id: ToolId
  title: string
  description: string
  icon: IconName
  route: `/${string}`
  availability: 'ready' | 'beta' | 'disabled'
  dataScope: 'local' | 'private-cloud' | 'public-cloud'
}

export const tools = [/* static manifests */] as const satisfies readonly ToolManifest[]
```

约束：

- 页面路径仍显式写入 `app.config.ts`；CI 校验 manifest route 全部存在且 id 唯一。
- manifest 不引用 React 页面组件，避免将所有页面代码拉入首页依赖图。
- 每个工具独立目录，禁止跨 feature 深层 import；共享能力提升到 `shared/`。
- “新增工具三步”改为生成器：创建目录与测试、写 manifest、写页面配置；随后由 CI 验证。

### 4.4 身份与权限

- 小程序启动后由平台身份建立用户上下文，不设置独立登录页。
- 只有执行私人云数据操作时才等待身份就绪；二维码和番茄钟不被登录阻塞。
- 不建立 `users` 集合，除非未来确实有用户级设置；openid 不是前端展示资料。
- `todos`、`favorites` 使用“仅创建者可读写”规则；客户端查询需匹配安全规则。
- `articles` 对客户端只读、仅云函数写；管理端密钥绝不进入小程序包。
- 权限规则作为仓库内配置或部署文档受版本控制，并有至少一组正向/越权测试。

### 4.5 数据模型

```ts
interface TodoRecord {
  _id: string              // client-generated UUID; retry-safe
  _openid: string          // platform-owned
  title: string
  completed: boolean
  createdAt: Date
  updatedAt: Date
  schemaVersion: 1
}

interface ArticleRecord {
  _id: string              // `${sourceId}:${externalId}` hash
  sourceId: string
  externalId: string
  title: string
  summary: string
  canonicalUrl: string
  publishedAt: Date | null
  fetchedAt: Date
  content?: SanitizedContent
  schemaVersion: 1
}

interface FavoriteRecord {
  _id: string              // hash(openid + articleId), idempotent
  _openid: string
  articleId: string
  createdAt: Date
  schemaVersion: 1
}
```

所有持久化结构带 `schemaVersion`。本地 Storage 键统一为 `bbl:<feature>:v<schema>`，读取时校验结构；损坏数据回退默认值并记录诊断，不让页面崩溃。

### 4.6 云函数

| 函数 | 触发 | 责任 |
|---|---|---|
| `syncReading` | 定时任务、管理端手动 | 拉取、校验、清洗、幂等写入内容 |
| `healthcheck` | 仅开发/部署验证 | 返回版本与依赖状态，正式版不暴露敏感信息 |

不创建只为返回 openid 的 `login` 函数，除非所选云开发接入方式经 P0 验证确实需要。普通待办 CRUD 优先走受安全规则保护的数据库；需要跨记录事务或服务端校验时才加云函数。

### 4.7 可观测性

- 错误统一为 `AppError { code, userMessage, retryable, cause }`。
- 日志只记录版本、工具 id、阶段、错误码和耗时，不记录待办标题、二维码内容、文章正文或 openid。
- 开发版设置页可复制诊断信息：应用版本、基础库、网络状态、最后同步时间、最近错误码。
- v0.1 至少观测：启动失败、页面加载失败、云端写失败、内容同步失败。

## 6. 质量方案

### 6.1 自动化测试

- 单元测试：番茄钟时间推导、状态迁移、数据校验、文章/收藏 id 生成。
- 组件测试：空状态、错误重试、写入防连点、待办撤销。
- 集成测试：repository 的成功、超时、权限拒绝和结构损坏。
- 构建检查：类型、lint、测试、manifest/route 一致性、生产包构建。

不追求笼统覆盖率数字；以下高风险纯逻辑必须 100% 分支覆盖：番茄钟状态机、计时恢复、幂等 id、持久化迁移。

### 6.2 真机矩阵

至少覆盖：一台较旧 iPhone、一台主流 Android、微信开发者工具。核心场景覆盖 Wi-Fi、蜂窝网络、断网恢复、拒绝相册权限、切后台 1 分钟与 30 分钟、杀进程重启。

H5 只能用于快速调样式和纯逻辑，不作为发布验收环境。Canvas、路由、生命周期、权限、云开发必须在微信小程序真机验证。

### 6.3 发布门槛

- 无 P0/P1 缺陷，P2 有明确绕行方案。
- 四个核心任务均通过真机验收；若阅读未满足内容源门槛，则按预案替换后再发布。
- 数据权限完成越权验证。
- 生产环境关闭调试日志和测试入口。
- 有回滚步骤：保留上一可用体验版、云函数版本和数据库变更说明。

## 7. 实施计划与止损点

| 阶段 | 产出 | 预计 | 退出标准 |
|---|---|---:|---|
| P0 风险验证 | Taro/原生对比 spike、Canvas、云身份/权限、内容源结论 | 1–2 天 | 四项决策都有真机证据 |
| P1 工作台骨架 | 首页、设置、manifest、生成器、CI、错误模型 | 1.5 天 | 空工具 30 分钟接入并通过 CI |
| P2 本地工具 | 番茄钟、二维码 | 1.5–2 天 | 后台恢复和三端扫码通过 |
| P3 私有云数据 | 待办、快照、权限规则 | 1.5–2 天 | 弱网/越权/重试测试通过 |
| P4 内容域 | 阅读同步、列表/详情/收藏 | 2–3 天 | 旧缓存回退、幂等同步通过 |
| P5 发布打磨 | 真机矩阵、性能、文案、隐私与回滚 | 1–2 天 | 满足发布门槛 |

预计 8.5–12 个工作日。旧方案的 8 天是单点估计，未包含技术 spike、真机兼容、权限测试、内容源合规和返工缓冲，不适合用作承诺。

关键止损：

1. P0 若 Taro 核心路径出现平台补丁，切原生，不继续沉没成本。
2. P0 若内容源未确认，阅读不进入 P4，替换为纯本地工具。
3. P3 若“离线写入”无法明确呈现一致性，保持只读快照，不临时扩展同步队列。
4. 任一工具超过 2 天仍未跑通核心任务，先缩范围再继续，不牺牲整体可发布性。

## 8. 风险清单

| 风险 | 概率/影响 | 预防 | 降级 |
|---|---|---|---|
| 框架与微信能力差异 | 中/高 | P0 真机 spike、锁版本 | 改微信原生 TS |
| 后台计时被暂停 | 高/中 | 绝对时间推导 | 回前台结算，不承诺后台通知 |
| 内容源变更或不允许转载 | 中/高 | 单源验证、保存最少内容 | 移除阅读或仅展示链接 |
| 云权限配置错误 | 中/高 | 规则入库、越权测试 | 暂停云工具入口 |
| 弱网造成重复写 | 中/中 | 客户端 id、写入锁、幂等设计 | 显示失败并手动重试 |
| QR 保存/扫码兼容 | 中/中 | 真机矩阵、固定导出规格 | 仅复制内容，暂关保存 |
| “插件化”过度设计 | 高/中 | 仅 manifest + 生成器 | 删除未产生复用的抽象 |
| 包体增长 | 中/中 | 逐项测量、避免整库引入 | 分包或替换依赖 |

## 9. 关键决策记录

| 决策 | 选择 | 原因 |
|---|---|---|
| 产品形态 | 个人工具工作台 | 与真实用户和低运营目标一致 |
| 首版工具 | 待办、番茄钟、二维码、阅读 | 覆盖主要价值与技术风险，去掉重复入口 |
| 扩展机制 | 构建期 manifest + 模板生成器 | 符合小程序静态路由约束，足够低成本 |
| 登录 | 无感平台身份 | 不收集无关资料，不打断本地工具 |
| 后端 | 微信云开发，按需 repository | 当前成本最低，不做虚假“一键迁移”承诺 |
| 离线 | 本地工具完整离线；云工具只读快照 | 范围可控，避免不可靠同步 |
| UI | 轻量自有组件 | 页面少，避免组件库兼容与体积成本 |
| 计时 | 绝对时间戳推导 | 生命周期暂停后仍可正确恢复 |
| 内容 | 定时服务端同步、单一已确认来源 | 控制稳定性、配额与合规风险 |

## 10. 开工前必须确认的四件事

这些不是继续写架构文档的问题，而是 P0 必须得到证据的产品/工程输入：

1. 微信小程序 AppID、云开发环境和体验成员是否已准备。
2. “阅读”准备接入哪个具体来源，是否允许缓存摘要或正文。
3. Taro 是已有偏好还是硬约束；若 spike 不通过是否接受原生 TypeScript。
4. 首版是否确实需要向熟人开放；若只供本人使用，可进一步删除部分空态引导和兼容工作。

四项确认后，本文即可拆成实施任务；确认前不应继续细化虚构的 API、组件或工期。
